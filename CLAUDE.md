# CLAUDE.md — SSV Network Web App

This file guides Claude Code when working with the SSV Network web application repository. Read this fully before making any changes.

## Project Overview

SSV Network is a decentralized Ethereum staking infrastructure using Secret Shared Validators (SSV/DVT). This repository is the **frontend web application** — the user-facing dApp for managing validators, operators, clusters, and staking on the SSV protocol.

**Current Release Target: v2.0.0 — "SSV Staking"**

This release introduces UI for three tightly coupled protocol upgrades:
1. **ETH Payments** — cluster fees denominated in ETH instead of SSV tokens
2. **Effective Balance (EB) Accounting** — fees scale with actual validator effective balance
3. **SSV Staking** — SSV holders stake tokens, receive cSSV, and earn ETH protocol revenue

## Build & Dev Commands


### Contract Interaction Pattern

ABIs live in `src/lib/abi/` (setter, getter, token). **Never edit ABI files manually** — they are auto-generated by `scripts/contracts/update-abi.js` from deployed contracts.

wagmi hooks are auto-generated from ABIs via `src/lib/contract-interactions/core/create-contract-hooks.ts` and output to `src/lib/contract-interactions/hooks/`. Run `pnpm generate:hooks` after any ABI update.

Contract addresses come from the network config at runtime via `useSSVNetworkDetails()`.

### Network Config (`VITE_SSV_NETWORKS`)

The app is built against a JSON array of network configs injected at build time. Each entry:

```ts
{
  networkId: number,           // Chain ID (e.g. 560048 for Hoodi)
  api: string,                 // Backend API base URL
  apiVersion: "v4",
  apiNetwork: "hoodi" | "mainnet",
  explorerUrl: string,
  insufficientBalanceUrl: string,
  tokenAddress: string,        // SSV ERC-20 address
  setterContractAddress: string, // SSVNetwork (write) proxy address
  getterContractAddress: string, // SSVNetworkViews address
  googleTagSecret?: string,
}
```

Validated with Zod at runtime in `src/hooks/use-ssv-network-details.ts`. Access via `useSSVNetworkDetails()` throughout the app.

### State Management

- **Valtio** — global mutable state (operators, validators, cluster wizard state)
- **Preact Signals** — fine-grained reactivity for frequently-updated values
- **TanStack Query** — server/chain state, caching, background refetch
- **React Hook Form + Zod** — form state and validation

## Environment Variables

All Vite env vars use the `VITE_` prefix and must be set at **build time**.

| Variable | Required | Description |
|---|---|---|
| `VITE_SSV_NETWORKS` | Yes | JSON array of network configs (see above) |
| `VITE_GAS_PRICE` | No | Override gas price (wei) |
| `VITE_GAS_LIMIT` | No | Override gas limit |
| `VITE_LINK_SSV_DEV_DOCS` | No | Dev docs URL |
| `VITE_EXPLORER_URL` | No | Block explorer URL |
| `VITE_DISTRIBUTION_CONTRACT_ADDRESS` | No | Token distribution contract |
| `VITE_MIXPANEL_TOKEN` | No | Mixpanel analytics token |
| `VITE_DEBUG` | No | Enable debug logging |
| `VITE_DISABLE_NETWORK_DATA_CHECK` | No | Skip network validation (dev only) |

Feature flags (uncomment to enable): `VITE_FAUCET_PAGE`, `VITE_CLAIM_PAGE`.

## CI/CD Pipeline (build_deploy.yml)

Three deployment environments, each triggered by a specific branch:

| Branch | Environment | URL |
|---|---|---|
| `stage` | Staging | app.stage.ssv.network |
| `main-preview` | Preview | preview.app.ssv.network |
| `temporary-hoodi-deploy` | Production (Hoodi) | app.ssv.network |

**Stage contract addresses** are defined once as job-level env vars (`STAGE_SETTER_CONTRACT_ADDRESS`, `STAGE_GETTER_CONTRACT_ADDRESS`) and referenced in both the `Set stage SSV networks config` step and `Check ABI compatibility` step.

The `Check ABI compatibility` step runs on every `stage` push and verifies local ABI files match the deployed contracts via `scripts/check-abi.js`.

## Working Branch & Git Workflow

- **Working branch**: `stage` (primary development branch)
- Create feature branches off `stage`, then PR back
- Semantic release runs on `stage` and `main` pushes
- Commit messages follow Conventional Commits (enforced by Commitizen)

## Code Generation Rules

- **Never manually edit** files in `src/lib/abi/` or `src/lib/contract-interactions/hooks/`
- After fetching new ABIs: run `pnpm generate:hooks` to regenerate wagmi hooks
- ABI update script: `node scripts/contracts/update-abi.js`

## Test Expectations

When writing tests:
- Use Vitest globals (`describe`, `it`, `expect`) — no imports needed
- Tests run in Node environment (not jsdom)
- Co-locate test files with source: `*.test.ts` / `*.test.tsx`
- For contract interaction hooks: follow patterns in `src/lib/contract-interactions/core/create-contract-hooks.test.ts`
